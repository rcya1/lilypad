Secure Multi-Party Computation

Suppose a set of nnn parties denoted by P1,…,PnP_1, \dots, P_nP1​,…,Pn​ with private inputs x1,…,xnx_1, \dots, x_nx1​,…,xn​ want to compute f(x1,…,xn)f(x_1, \dots, x_n)f(x1​,…,xn​) without rebealing anything about each of their inputs

  • This is the goal of MPC

Examples:

  • Collaborating on financial / healthcare / market research without rebealing each of their inputs
  • Machine learning without sharing raw data
  • Secure auctions

We assume that every pair of parties has a private and authenticated channel

  • We assume the network is synchronous
    • Protocols proceeds in rounds, and at the end of each round, all messages sent were received

Defining security:

  • Take 1:
    • By the end of the protocol, no party learns any information about the others' inputs
    • This is impossible since everyone learns f(x1,…,xn)f(x_1, \dots, x_n)f(x1​,…,xn​)
  • Take 2:
    • By the end of the protocol, no party learns any information about the others' inputs besides the output fff
    • This doesn't provide enough security because we want even more
      • Let's say we were implementing an auction
      • We don't want it to be somehow possible for someone else to be able to guarantee outbid someone by 1 even if they don't know that person's bid
  • Take 3:
    • The MPC protocol should be as secure as an idealized world where everyone sends their result to a trusted third party and they compute the output
    • We formalize this below
Security

An nnn party protocol securely computes a function in the presence of at most ttt corruptions if for every PPT adversary A\mc{A}A controlling at most ttt parties, there exists a PPT simulator S\mc{S}S controlling the same parties in the ideal world such that for any inputs, RealA\t{Real}_\mc{A}RealA​ and IdealS\t{Ideal}_\mc{S}IdealS​ are indistinguishable

  • RealA\t{Real}_\mc{A}RealA​ refers to the output of all parties after running the protocol where the adversarial parties output whatever they want
  • IdealS\t{Ideal}_\mc{S}IdealS​ is the same story but they all hand their inputs to a secure third-party

There are two main types of adversaries:

  • Malicious: the adversary controlling the parties completely controls them and sends whatever the adversary wants
  • Honest-but-curious: an adversary controlling the parties learns their inputs and all messages sent / received, but cannot modify the messages

BGW Protocol

The BGW protocol consists of three phases:

  1. Secret Sharing of Inputs
  2. Computation on Shares
  3. Reconstruction

Secret Sharing of Inputs

Fix a prime p>np > np>n. Each party then uses Shamir's secret sharing scheme to share their input across t+1t + 1t+1 parties:

  • Select uniformly at random ttt coefficients a1,…,ata_1, \dots, a_ta1​,…,at​ from GF[p]\t{GF}[p]GF[p] and let:
    gb(x)=b+a1x+⋯+atxtg_b(x) = b + a_1x + \dots + a_tx^tgb​(x)=b+a1​x+⋯+at​xt
  • Send gb(j)g_b(j)gb​(j) to party jjj

So now, any subset of t+1t + 1t+1 parties can reconstruct any other parties' input by working together

Computation on Shares

We think of the function fff as some combination of addition and multiplication gates

For addition gates:

  • Suppose the inputs to a gate are aaa and bbb
  • These are shared via polynomials gag_aga​ and gbg_bgb​
  • We need to compute the iii th share of ga+bg_{a + b}ga+b​
  • We note that the definition of ga+bg_{a + b}ga+b​ is a random polynomial such that ga+b(0)=a+bg_{a+b}(0) = a + bga+b​(0)=a+b
    • We note that ga+b=ga+gbg_{a+b} = g_a + g_bga+b​=ga​+gb​ works
    • Therefore we can compute the iii th share of ga+bg_{a + b}ga+b​ by ga+b(i)=ga(i)+gb(i)g_{a + b}(i) = g_{a}(i) + g_b(i)ga+b​(i)=ga​(i)+gb​(i)

For multiplication by a constant:

  • We can also just use gca=c∗gag_{ca} = c * g_agca​=c∗ga​

For multiplication gates, it is more complicated:

  • We cannot just use gab=gagbg_{ab} = g_a g_bgab​=ga​gb​
    • This is degree 2t2t2t !
    • We would need 2t+12t + 12t+1 parties to reconstruct, and every element is not random because it is the product of two polynomials
  • We do two steps to fix this:
    • Degree reduction
    • Rerandomization

Degree Reduction: Each party reduces it share of a degree 2t2t2t polynomial to a share of some degree ttt polynomial

Rerandomization: The problem is this polynomial might not be very random anymore! So we rerandomize the degree 2t2t2t polynomial before truncating it

  • Each party PiP_iPi​ secret shares the value 000 using a 2t+12t + 12t+1 secret sharing scheme
    • Namely, each party PiP_iPi​ finds a random 2t2t2t polynomial gig_igi​ such that gi(0)=0g_i(0) = 0gi​(0)=0

  • This is the polynomial that is then passed into degree reduction

Reconstruction

The parties then each send their share of the output to whomever should receive said output

  • These parties then reconstruct

Security

Intuitively, each party holds random shares and an adversary cannot reconstruct any input with only ttt controlled parties

  • Note that this doesn't rely on cryptography!
  • It is information theoreticlly secure

What happens if we have malicious parties who send random junk?

  • Recall from Shamir's secret sharing scheme that we can reconstruct even if n−t−12\frac{n - t - 1}{2}2n−t−1​ parties are corrupted
  • This means that as long as t<n/3t < n/3t<n/3, then we can reconstruct!
  • If the adversary only sends junk on the reconstruction phases, then we are good
  • If the adversary sends junk on the secret sharing phase, then it is no longer secure
    • I.e. if it sends things that do not correspond to a degree ttt function
    • This is fixable if we use a verified secret sharing scheme
      • A dealer "proves" that it send shares corresponding to a degree ttt polynomial (or degree 2t2t2t in rerandomization)
      • If we use one of these, then we are fully secure against t<n/3t < n/3t<n/3 adversaries
